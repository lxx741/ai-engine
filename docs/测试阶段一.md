### 8.3 失败原因分析（深度分析）

**主要问题**：API 路由返回 404（52 个失败用例）

**已验证的事实**：
- ✅ 环境变量已正确设置（`API_PREFIX=/api` 在 global-setup 中输出确认）
- ✅ ConfigModule 配置正确（使用 load 提供默认值）
- ✅ main.ts 使用 process.env 回退
- ✅ AppTestModule 硬编码配置为 `/api`
- ❌ 所有涉及 `/api/apps` 和 `/api/workflows` 的测试都返回 404
- ✅ models、tools、chat 测试通过（这些端点可能是公开的或使用不同路由）

**可能的根本原因**：

1. **ApiKeyGuard 全局守卫问题** ⭐ 最可能
   - `@UseGuards(ApiKeyGuard)` 在 Controller 级别应用
   - 测试中 Mock 了 Service 但未 Mock Guard
   - Guard 验证失败导致 404（而非 401/403）

2. **ThrottlerModule 配置问题**
   - `ThrottlerModule.forRoot()` 在测试中可能未正确初始化
   - 限流守卫可能拦截请求

3. **路由前缀实际未应用**
   - ConfigModule 的 load 配置可能在模块编译后被忽略
   - `app.setGlobalPrefix()` 在 TestingModule 中可能不生效

**证据分析**：
```
✅ models.e2e-spec.ts 通过 - /api/models 端点使用 @Public() 装饰器
✅ tools.e2e-spec.ts 通过 - /api/tools 端点使用 @Public() 装饰器  
✅ chat.e2e-spec.ts 通过 - 使用了完整的 Prisma Mock
❌ apps.e2e-spec.ts 失败 - /api/apps 端点有 ApiKeyGuard
❌ workflows.e2e-spec.ts 失败 - /api/workflows 端点有 ApiKeyGuard
```

**结论**：问题很可能在于 **ApiKeyGuard 在测试环境中未正确 Mock**，导致认证失败返回 404 而非预期的 200/201。

### 8.4 最终解决方案（✅ 已实施）

**实施步骤**：
1. ✅ 创建 `test/app-test.module.ts` 使用 Mock Service
2. ✅ 使用 Injection Token 解决依赖注入问题
3. ✅ 在测试中调用 `app.setGlobalPrefix('/api')`
4. ✅ 不导入 AuthModule 和 ThrottlerModule（避免 Guard 干扰）
